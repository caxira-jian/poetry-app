# 古诗背诵助手 V1 当前实现说明（需求文档版）

## 1. 文档目的
- 说明当前线上版本的实现边界、行为与配置方式。
- 作为后续产品迭代的基线文档，避免“目标需求”和“已实现状态”混淆。

## 2. 产品目标（当前版本）
- 手机优先的古诗背诵 Web 应用。
- 支持本地诗库管理、背诵记录、LLM 解析口语输入、LLM 推荐。
- 支持访客可用的默认 API（服务端托管）和用户自定义 API（本地加密）。
- 支持本地数据导入导出（JSON）。

## 3. 页面与核心流程
### 3.1 页面
- 今日（首页）：生成推荐（复习/新学）。
- 诗库：口语输入解析（新增/修改/删除/意向调整）+ 预览确认。
- 记录：口语输入解析为背诵记录（完成/熟练）+ 预览确认。
- 模型：API 模式切换、主密码管理、模型配置、连通性测试。
- 数据：种子诗库初始化、JSON 导入/导出。

### 3.2 用户主流程
1. 初始化种子诗库或导入 JSON。
2. 在“诗库/记录”用口语输入，LLM 解析为结构化动作。
3. 用户确认后入库（会展示新增/删除/修改汇总）。
4. 首页点击“生成今日推荐”，得到复习和新学习建议。

## 4. 领域模型与数据结构
### 4.1 Poem
- `id`, `title`, `author`, `content`, `dynasty?`, `tags[]`
- `learnIntent`: `known | learning | wishlist`
- `currentStatus`: `none | completed | proficient`
- `masteryLevel`: `0-5`
- `lastRecitedAt?`: ISO8601

### 4.2 ReciteLog
- `id`, `poemId`
- `titleSnapshot`, `authorSnapshot`
- `recitedAt`: ISO8601
- `status`: `completed | proficient`
- `note?`

### 4.3 ProviderConfig
- `provider`: `qwen | deepseek | glm | custom`
- `baseUrl`, `model`, `enabled`, `temperature`
- `encryptedApiKey?`, `encryptedApiKeyIv?`, `keyAlias?`

### 4.4 RecommendationResult
- `review: Array<{ poemId, reason }>`
- `newLearning: Array<{ poemId, reason }>`
- `source: llm | fallback-list`

## 5. 存储设计
- 主存：IndexedDB
  - 表：`poems`, `recite_logs`, `settings`
- 推荐和列表查询使用诗词快照字段（如 `currentStatus`, `lastRecitedAt`）。
- 背诵事件采用日志累积，诗词表维护当前快照状态。

## 6. LLM 交互设计
### 6.1 提示词管理
- 所有前端 LLM 提示词集中在：`shared/llmPrompts.ts`
- 包含：
  - 口语解析系统提示词
  - 推荐系统提示词
  - 推荐与口语解析的用户模板提示词
  - 连通性测试最小消息（`ping`）

### 6.2 交互场景
- 口语输入解析（诗库/记录）：LLM 输出结构化 `actions`，支持预览与确认。
- 今日推荐：LLM 输出 `review/newLearning`。
- 模型连通性测试：最小请求检查 API 可达性。

### 6.3 失败降级
- 推荐失败时降级到本地规则候选列表（`fallback-list`）。
- 解析失败时保留错误并不入库。

## 7. API 模式与安全
### 7.1 默认 API 兜底（访客可直接用）
- 前端调用 `/api/llm`。
- 服务端读取 `DEFAULT_API_*`，再转发到模型厂商接口。
- 前端不持有 `DEFAULT_API_KEY`。

### 7.2 自定义 API（用户 key + 模型）
- 用户在浏览器本地配置 provider/model/key。
- key 使用主密码派生密钥后本地加密保存。
- 错误主密码无法解密。

### 7.3 主密码体验
- 支持会话解锁。
- 支持“记住浏览器 1 天”，减少重复输入。

## 8. 服务端代理（`/api/llm`）当前能力
- 支持 `action: test | chat`。
- 默认限流控时（已实现）：
  - `DEFAULT_API_CHAT_MAX_TOKENS`（默认 600）
  - `DEFAULT_API_TIMEOUT_MS`（默认 12000）
- Qwen 场景自动添加 `enable_thinking=false` 以减少 reasoning 带来的高延迟。
- 记录详细日志：
  - `requestMessages`（请求消息）
  - `responseContent`（模型回答文本）
  - `upstreamBody`（上游完整 JSON）
  - `usage`（token）
  - `timings`（分段耗时）

## 9. 日志与可观测性
### 9.1 本地日志
- 默认：`logs/llm-latency.log`
- 可配置：
  - `LLM_LOG_DIR`
  - `LLM_LOG_FILE`

### 9.2 Vercel 日志
- 在 Functions 日志中搜索 `llm-latency`。
- 可查看每次请求的 prompt、回答、token、耗时。

### 10.1 必需（默认 API）

- `DEFAULT_API_MODEL`

### 10.2 推荐
- `DEFAULT_API_PROVIDER`

## 11. 部署说明（Vercel）
1. GitHub push 到 `main` 自动触发构建。
2. 在 Vercel 配置 `DEFAULT_API_*` 环境变量。
3. 变更环境变量后需 Redeploy 才生效。

## 12. 已知约束（当前版本）
- 无账号体系、无多端同步（数据只在当前设备）。
- LLM 输出质量受模型与提示词约束，仍可能出现空结果或格式偏差。
- 推荐逻辑对诗库规模较大时，提示词与调用成本会明显上升。

## 13. 下一步改进建议（按优先级）
1. 推荐输入分层抽样：减少发送给模型的诗词规模，继续降延迟与成本。
2. 为 `/api/llm` 增加结构化错误码与重试策略（区分超时、余额、格式错误）。
3. 新增“模型对话调试页”（仅管理员可见），可直接查看最近 N 次 prompt/response。
4. 增加自动化测试：
   - NLU 动作解析与入库一致性
   - 推荐返回结构校验
   - JSON 导入导出完整性
5. 规划账号体系与云端同步（若进入多设备场景）。
